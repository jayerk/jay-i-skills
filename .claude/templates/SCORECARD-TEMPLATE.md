## How to Use This Scorecard

1. **Set context.** Fill in the metadata above — lifecycle stage and product type determine your target levels.
2. **Score each sub-practice.** For each row in the assessment table, read the maturity rubric in `knowledge/product-standards.md` and pick the level that best matches your team today. Use evidence, not aspiration.
3. **Look up targets.** Use the Target Reference Table below to find the target level for your lifecycle stage. Fill in the Target column.
4. **Calculate gaps.** Gap = Target − Current. Positive gaps are improvement opportunities. Zero means you're on target. Negative gaps at Sunset are normal — the target dropped because scope narrowed.
5. **Prioritize actions.** Start with the largest gaps in the lowest adoption wave. Wave 1 foundations unlock Waves 2-4 — fixing a Wave 3 gap while Wave 1 is weak creates dependency debt.

**How long this takes:** 30-60 minutes for a first assessment using the sub-practice summary. 90-120 minutes if using the standard-level detail section.

**Who should fill this in:** Product manager or product lead, ideally with input from design, engineering, and delivery leads. Self-assessment works — it's a conversation starter, not a performance review.

## Scoring Guide

### How to Determine Your Current Level

For each sub-practice, read the maturity rubric in `knowledge/product-standards.md`.
Pick the level where your team matches **most** of the observable behaviors. If
you're between levels, round down — the rubric measures what's consistently in
place, not occasional good behavior.

| Level | Harvey Ball | What It Means |
|:---:|:---:|---|
| 1 | ○ | Emerging — practice doesn't exist or is ad-hoc |
| 2 | ◔ | Developing — some elements exist but inconsistent |
| 3 | ◑ | Established — practice is in place and working |
| 4 | ◕ | Advanced — optimizing, calibrating, extending |
| 5 | ● | Leading — teaching others, evolving the practice |

### How to Read the Gap

| Gap | Meaning | Action |
|:---:|---|---|
| +3 or more | Critical gap — foundational practice missing | Start here. Use the Adoption Path from `product-standards.md`. |
| +2 | Significant gap — practice exists but isn't working | Prioritize this cycle. Likely a Wave 1-2 issue. |
| +1 | Minor gap — practice is close to target | Address when capacity allows. |
| 0 | On target — practice meets expectations for this stage | Maintain. Consider whether to push toward next level. |
| −1 or less | Below target (Sunset) — appropriate scope reduction | Not a gap. Target dropped because scope narrowed. |

### Prioritization Logic

When multiple gaps are equal, prioritize by **adoption wave** (column 4 in the
assessment table):

1. **Wave 1 — Foundations:** CMI, PEI, MO. Fix these first — they feed everything downstream.
2. **Wave 2 — Structure:** S&P, S&R. Turn foundations into actionable frames.
3. **Wave 3 — Action:** CEJ, VBS. Translate understanding into prioritized work.
4. **Wave 4 — Governance:** VGR, ADP, RfA. Operationalize the system.

Fixing a Wave 3 gap while Wave 1 has a bigger gap creates **dependency debt** —
surface-level compliance without real capability. See
`knowledge/product-standards-dependencies-adoption.md` for the full rationale.

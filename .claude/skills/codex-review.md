# codex-review

Verification layer — hands off Claude's output to OpenAI Codex for independent
review. Claude builds, Codex reviews. Cross-context: used by any skill that
produces code, structured artifacts, or testable output.

TRIGGER when: user asks to "verify this with Codex", "have Codex review",
"run a Codex check", or when a skill produces code or structured output
that benefits from independent verification.

DO NOT TRIGGER when: output is pure prose (use writing-triage instead),
user is asking about Codex conceptually, or the repo has no AGENTS.md
configured.

---

## The Model

Two agents, two roles, one repo:

| Agent | Role | Strengths |
|-------|------|-----------|
| Claude Code | **Builder** — writes code, creates artifacts, implements features | Deep context from CLAUDE.md, skills, personas, frameworks |
| Codex CLI | **Reviewer** — verifies assumptions, runs tests, catches blind spots | Independent perspective, sandbox execution, test-driven verification |

Claude never reviews its own work for correctness. Codex never builds from
scratch without Claude's context. The handoff is the quality gate.

---

## Prerequisites

1. **Codex CLI installed** — `npm i -g @openai/codex` or `brew install --cask codex`
2. **AGENTS.md present** — in the target repo root (generated by `repo-init` or manually)
3. **Authentication** — Codex authenticated via ChatGPT account or API key

If prerequisites aren't met, tell the user what's missing and stop.

## Review Types

### 1. Code Review

After Claude writes or modifies code, hand off to Codex for review.

**What to ask Codex:**
- Review the diff for logic errors, edge cases, and security issues
- Check that the implementation matches the stated intent
- Flag anything that looks like an assumption rather than a verified fact

**Invocation pattern:**
```
codex "Review the changes in [file(s)]. Check for logic errors, edge cases,
and whether the implementation matches this intent: [one-sentence intent].
Flag assumptions."
```

### 2. Test Verification

After Claude writes tests or claims something is testable, have Codex run them.

**What to ask Codex:**
- Run the test suite and report results
- Check for missing test coverage on the changed code
- Verify that tests actually test what they claim to test

**Invocation pattern:**
```
codex "Run the tests for [scope]. Report any failures. Check if the tests
cover the changes in [file(s)] and flag gaps in coverage."
```

### 3. Assumption Check

When Claude's output depends on assumptions about the codebase, APIs, or
behavior, have Codex verify them independently.

**What to ask Codex:**
- Verify that the assumed API behavior is correct
- Check that referenced files/functions exist and have the expected signatures
- Confirm that the stated dependencies are actually available

**Invocation pattern:**
```
codex "Verify these assumptions about the codebase: [list assumptions].
Check actual file contents, function signatures, and dependencies.
Report what's confirmed and what's wrong."
```

### 4. Integration Check

Before merging or shipping, have Codex do a full integration pass.

**What to ask Codex:**
- Build the project and report errors
- Run the full test suite
- Check for import/dependency issues introduced by the changes

**Invocation pattern:**
```
codex "Build the project and run the full test suite. Report any build
errors, test failures, or dependency issues. Focus on changes in [branch/files]."
```

## Workflow

### Step 1 — Claude completes the build

Claude finishes writing code, tests, or structured artifacts. Before
committing, Claude summarizes:
- **What changed** — files modified, functions added/changed
- **Intent** — what the change is supposed to accomplish
- **Assumptions** — anything Claude assumed but didn't verify

### Step 2 — Hand off to Codex

User runs the appropriate Codex invocation (see Review Types above).
Claude provides the exact command to copy-paste, pre-filled with
the specific files, intent, and assumptions from Step 1.

### Step 3 — Process Codex feedback

When Codex returns results:
- **Green (no issues)** — proceed to commit
- **Yellow (suggestions)** — Claude evaluates and incorporates where valid
- **Red (failures/errors)** — Claude fixes the issues, then re-submits to Codex

### Step 4 — Document the review

Add a one-line note to the commit message:
```
Codex-reviewed: [review type] — [outcome: clean / fixed N issues]
```

## AGENTS.md

Codex uses `AGENTS.md` files the way Claude uses `CLAUDE.md` — repo-level
instructions that shape how the agent operates. Every repo in the jay-i-skills
ecosystem should have an AGENTS.md that tells Codex:

1. **What this repo is** — one sentence
2. **How to build and test** — exact commands
3. **What to focus on during review** — repo-specific quality concerns
4. **What to ignore** — generated files, vendored dependencies, etc.

Use the `repo-init` skill to generate AGENTS.md alongside CLAUDE.md for
downstream repos. See `.claude/templates/AGENTS-TEMPLATE.md` for the format.

---

## Quality Checks

- **Independence** — Codex must review without Claude's reasoning context.
  The handoff includes *what* and *intent*, not Claude's chain of thought.
- **Specificity** — every Codex invocation names specific files and a clear
  intent. Never "review everything" — always scoped.
- **Actionability** — Codex feedback must result in a concrete action:
  fix, accept, or document why the finding doesn't apply.
- **Traceability** — commit messages record that Codex review happened
  and what the outcome was.

---

## When NOT to Use Codex Review

- Pure prose output — use `writing-triage` instead
- Exploratory brainstorming — neither agent should review ideation
- Trivial changes — single-line fixes, comment edits, formatting
- When Codex CLI is not installed or authenticated

---

## Persona Note

This skill operates above all three context personas. It is infrastructure
for the build-review handoff pattern — voice should be direct and precise.
The goal is verification confidence, not ceremony.
